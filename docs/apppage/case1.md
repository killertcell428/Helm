# Case 1: 経営会議後の構造変化検知

## 概要
NeoWave Communications（通信系IT大手）の経営会議を題材にしたデモケース。業績悪化の認識は共有されているが、戦略変更の議論が進まない「正当化フェーズ」を構造的問題として検知する。

## ページ構成

### 1. ナビゲーションバー
- **Helmロゴ**: CompassアイコンとHelmブランド名
- **状態リセットボタン**: セッション状態をリセット
- **戻るボタン**: 「デモ一覧に戻る」ボタン（/demoへリンク）

### 2. ヘッダー
- **バッジ**: 「Case 1」
- **タイトル**: 「経営会議後：戦略見直しの構造的先送り」
- **説明**: NeoWave Communications（通信系IT大手）の経営会議。業績悪化の認識は共有されているが、戦略変更の議論が進まない。Helmはこの「正当化フェーズ」を構造的問題として検知します。
- **現在の状態バッジ**: 現在のPDCAサイクルの状態を表示

### 3. AIの裏側サマリ
- **説明**: 現在の状態に応じて、HelmのAIが何をしているかを表示

## 状態遷移（PDCAサイクル）

### State 1: Before（人だけの場合）
Helm導入前の状態を表示。

#### 指標（Helm導入前の状況）
- **意思決定リードタイム**: 70-85日（様子見が継続）
- **正当化フェーズ検知率**: 0%（検知不可能）
- **代替案検討率**: 10%未満（撤退選択肢なし）

#### 経営会議議事録
- **会議名**: 四半期経営会議
- **参加者**: CFO, CEO, 通信本部長, DX本部長
- **主な発言**:
  - CFO: モバイルARPUは前年同期比▲6.2%、5G設備投資は当初計画比＋18%、DX事業は売上成長しているが利益率は依然▲12%
  - CEO: 「厳しいが、我々の戦略自体は間違っていないと思う」
  - 通信本部長: 「ARPU低下は市場要因も大きい。短期では打てる手が限られる」
  - DX本部長: 「今は仕込みのフェーズ。ここで引くと将来がなくなる」
- **結論**: 2025年度計画は維持。各本部はコスト最適化を検討。次回進捗報告は3か月後

#### 会議後チャット
- **チャンネル**: 経営企画チャンネル
- **主な発言**:
  - 経営企画A: 「数字かなり厳しかったですね…」
  - 経営企画B: 「ですね。ただCEOも今回は触れなかったので、深掘りは次回かなと」
  - 通信本部補佐: 「『撤退』とか言い出す空気ではなかったですね」

#### よくある組織課題
- 「様子を見る」判断が継続し、戦略見直しの機会を逃す
- 「やめる選択肢」が構造的に議論されず、プロジェクトが継続
- 意思決定の先送りにより、組織リソースの最適配分が困難に

#### デモパターン選択
3つのパターンから選択可能：
- **パターンA**: 典型的な「正当化フェーズ」強め。KPI下方修正が複数回あるにも関わらず、撤退・ピボット議論が行われない。
- **パターンB**: 撤退・ピボット議論あり。健全な議論が行われており、「やめる選択肢」が構造的に検討されている。
- **パターンC**: リスクは出ているが報告・エスカレーションが曖昧。チャットでリスク提起があるが、会議内でははっきりエスカレーションされていない。

### State 2: データ受領直後
Helmがデータを受領した状態。

#### 自動取り込み完了
- 経営会議議事録、CFOレポート、Slackチャットログが自動的にHelmに取り込まれる
- 構造分析を開始

#### 元のデータを確認
- Google Meetへのリンク
- Google Chatへのリンク

### State 3: Helm解析完了
Helm構造診断が完了した状態。

#### 定量評価結果
- **総合スコア**: 70点台（ルールベース + マルチ視点LLMのアンサンブル）
- **重要度**: HIGH
- **緊急度**: MEDIUM
- **検出パターン数**: 構造的問題の数

#### スコアの見方
- ルールベース評価と、Executive / Corp Planning / Staff / Governanceなど複数ロールの視点別スコアをアンサンブルして算出
- 構造的なリスクが強い場合は、視点別スコアよりも総合スコアが抑えめに出ることがある

#### 視点別の評価
各ロール（Executive, Manager, Staff等）の視点から評価したスコアと説明を表示

#### Helm Finding
検出された構造的問題パターン：
- **パターンID**: B1_正当化フェーズ（例）
- **説明**: 詳細な説明文
- **重要度・緊急度**: バッジで表示
- **評価理由**: 複数の理由をリスト表示
- **証拠**: 会議やチャットから抽出された証拠
- **定量的メトリクス**:
  - KPI下方修正回数
  - 判断集中率
  - 反対意見無視件数

### State 4: Executiveの判断
Executiveが判断を行う状態。

#### Helmからのエスカレーション
- **対象ロール**: Executive
- **理由**: エスカレーションが必要な理由を説明

#### Helmからの提案
次回経営会議に以下を強制議題として追加することを推奨：
1. 「やめる選択肢を含めた3案比較」を議題化
2. 「撤退案の提案責任者」を明示的に任命
3. KPIトリガー（ARPU▲X%で再検討）を設定

#### Executiveの判断選択
- **了承する**: 提案をそのまま承認
- **一部修正して実行**: 提案を修正して実行

### State 5: 次アクション確定
Executiveの判断が反映された状態。

#### 確定されたアクション
- 次回経営会議に「やめる選択肢を含めた3案比較」を強制議題化
- 撤退案の提案責任者を明示的に任命
- KPIトリガー（ARPU▲X%で再検討）を設定

#### これからAIが自律的に実行
判断が確定したため、HelmのAIエージェントが自律的に具体的なアクションを実行。人の介入が必要な場面でのみ、再度停止する。

### State 6: AI自律実行中
AIが自律的にタスクを実行している状態。

#### Helmが自律的に実行しているタスク
1. **市場・社外データのリサーチ**（進捗20%）
   - Google Workspace連携のリサーチエージェントが、業界レポートや公開情報からARPU推移・競合動向などのサマリーを取得

2. **社内データ・過去案件の統合**（進捗40%）
   - 社内の過去案件・KPIのメタデータを集約し、現在の会議ログ/チャットと突き合わせて類似ケースとリスクパターンを抽出

3. **3案比較（継続・縮小・撤退）のシミュレーション**（進捗60%）
   - 継続・縮小・撤退それぞれの売上・コスト・利益とリスクレベルを試算し、3案比較用の下書きデータを生成

4. **資料ドラフト作成と関係者への通知準備**（進捗80%）
   - 3案比較の内容をもとにGoogle Docs形式の資料ドラフトを生成し、CFOや各本部長向けの事前共有メッセージ（下書き）を組み立て

5. **次回会議のアジェンダ案作成**（進捗100%）
   - 「やめる選択肢を含めた3案比較」を次回経営会議の正式議題候補として整理し、カレンダー/会議体への反映案を作成

#### 実行方針の説明
- データ収集・シミュレーション・資料ドラフト作成まではAIが自律的に実行
- 実際の招集・送信・最終決裁など、人の権限が必要な部分ではHelmが自動的に停止し、Executiveに確認を求める

### State 7: 実行結果受領
AI実行の結果を受領した状態。

#### 完了したタスク
- **分析レポート**: 3案比較完成（財務シミュレーション含む）
- **関係者通知**: 8名に配信完了（事前レビュー依頼済み）
- **会議設定**: 議題追加済み（次回経営会議：3週間後）
- **KPIモニタリング**: 自動追跡開始（トリガー条件設定完了）

#### 3週間後の経営会議での変化
- 撤退案スライドが議題の1つとして自然に含まれている
- CFOが「投資凍結オプション」の財務影響について具体的に言及
- KPIトリガーが明文化され、次回以降の判断が迅速化

#### 生成された資料
- Google Docs、Google Sheets、Google Driveへのリンク
- 各資料の閲覧・編集リンク

### State 8: 次のPDCAサイクルへ
3ヶ月後：PDCAサイクル継続中の状態。

#### Helm導入後の効果指標（フェルミ推定）
- **意思決定リードタイム**: 2日以内（導入前：70-85日から97%削減、35倍改善）
- **正当化フェーズ検知率**: 90%以上（導入前：0%）
- **代替案検討率**: 50%以上（導入前：10%未満から5倍以上改善）
- **構造スコア改善**: 75点 → 40点（47%改善）

#### 組織構造が継続的に改善
Helmは次の経営会議データを受領し、再び構造分析を実施。今回は「撤退案」が自然に議論され、意思決定スピードが向上。

#### Helmが実現したこと
「AIが止まるのは、判断が必要な時だけ。」それ以外は自律的にデータ収集、分析、関係者連絡、資料作成を実行。組織は「何をすべきか」ではなく「なぜ決まらないか」という構造的問題に集中でき、PDCAサイクルが自然に回り続ける。

#### 次のサイクル
Helmは再び会議データを受領 → 構造分析 → 必要なら新しい介入を提案。組織は学習し続け、時間とともにより賢くなっていく。

## インタラクション
- パターン選択: Before状態で3つのパターンから選択
- 状態遷移: 「次へ」ボタンで各状態を順次遷移
- Executive判断: 「了承する」「一部修正して実行」を選択
- 状態リセット: ナビゲーションバーからセッション状態をリセット可能
- 成果物確認: 実行結果受領状態で成果物の詳細ページ（/demo/case1/deliverables）へリンク

## 技術仕様
- 状態管理: localStorageに状態を保存（24時間有効）
- WebSocket: 実行進捗をリアルタイムで監視
- フォールバック: WebSocket接続失敗時はポーリング方式に切り替え

## Helm 3パターン PDCA デモ用ユースケース設計

### 共通前提

- 対象: Biz新規事業領域のプロジェクトチーム
- 共通メトリクス:
  - 判断集中度（decision_concentration_rate）
  - KPI目標の下方修正回数（kpi_downgrade_count）
  - 反対意見が無視された回数（ignored_opposition_count）※簡易指標
- 共通Findingsパターン:
  - R1_judgment_delay: 「判断の先送り・判断集中」
  - B1_justification_phase: 「正当化フェーズ（Zombie Project化の兆候）」
  - 追加予定: R2_escalation_gap, P1_planning_authority_mismatch（3か月版で拡張）

---

## Case 1: 経営会議後の議事録＋資料＋分析Excel

### 状況（ナラティブ）

- 新規事業の四半期ごとの経営会議。
- 分析担当からは「成長率が計画を下回り、CACが悪化している」と報告。
- 会議では、
  - KPI目標を「今期はブランド投資フェーズ」として下方修正
  - 「もう1四半期だけ様子を見る」という結論
- 誰も「撤退 or ピボット」を正式議題として挙げず、
  - 「続ける理由」だけが増えていく状態。

### Run #1（会議直後 / Before）

- メトリクス（例）:
  - decision_concentration_rate ≒ 0.8（経営会議の最終判断が常に同じ役員）
  - kpi_downgrade_count = 3
  - ignored_opposition_count = 1（分析担当の慎重意見が議事録に残るが、決定に反映されていない）
- 時系列メトリクス（例）:
  - kpi_downgrade_trend = 増加（過去4週間でKPI下方修正が3回発生、前回比+2回）
- 組み合わせパターン（例）:
  - パターン3: Zombie Project化の兆候
    - 条件: `kpi_downgrade_count >= 3` AND `B1_justification_phase` AND `decision_concentration_rate >= 0.8`
    - 組み合わせスコア = 0.85（閾値0.7を超過）
- Findings（期待）:
  - B1_justification_phase（重大）
  - R1_judgment_delay（中〜高）
- Helm視点:
  - 「事業の状態」ではなく、「会議の構造」が壊れ始めている。
  - KPI目標を下げ続けることで、撤退ラインの議論が起きない構造。
  - **時系列評価**: 過去4週間でKPI下方修正が増加傾向にあり、構造的問題が進行している。
  - **最小シグナル**: 3つのメトリクス（KPI下方修正、正当化フェーズ、判断集中）が同時に閾値を超えており、Zombie Project化の兆候が明確。

### Run #2（Helm提案反映後 / After）

- 介入内容（アイデア）:
  - 次回会議アジェンダに「撤退・ピボット条件」を必須項目として追加。
  - 分析担当を意思決定メンバーに昇格（Reviewer 的役割）。
- メトリクス（理想像）:
  - decision_concentration_rate ≒ 0.6（役員 + 分析担当の合議）
  - kpi_downgrade_count = 3（増えてはいない）
  - ignored_opposition_count = 0
- Findings（期待）:
  - B1_justification_phase は継続だが severity が中程度に緩和、もしくは消失へ向かうトレンド。

## Case 2: 重大ミスのエスカレーションチャット

### 状況（ナラティブ）

- スタッフが重大なオペレーションミスをし、重要なキャンペーンが遅延 or 実施内容の変更が必要に。
- チャット上で、
  - 現場メンバー間では「ヤバい」と共有されている。
  - しかし「誰がどこまで上長に上げるか」「いつまでに決めるか」が曖昧。
- 結果として、
  - 対応が後手に回り、ブランドリスクや機会損失が拡大。

### Run #1（チャット後 / Before）

- メトリクス（例）:
  - decision_concentration_rate ≒ 0.4（現場メンバー間で分散）
  - kpi_downgrade_count = 0 or 1（まだ数字には反映されていない）
  - ignored_opposition_count = 2（リスクを強く訴えるメッセージがスルーされている）
- 時系列メトリクス（例）:
  - opposition_suppression_trend = 増加（過去4週間で反対意見無視が2回発生、前回比+1回）
- 組み合わせパターン（例）:
  - パターン2: エスカレーション遅延の兆候
    - 条件: `ES1_報告遅延` AND `decision_concentration_rate < 0.5` AND `ignored_opposition_count >= 2`
    - 組み合わせスコア = 0.75（閾値0.7を超過）
- Findings（期待）:
  - R2_escalation_gap（責任境界のギャップ）※新パターン
    - 「現場レベルではリスク認識があるが、エスカレーションルールが不在」
- Helm視点:
  - ミスそのものではなく、「リスク情報が上がりきらない構造」を問題視。
  - **時系列評価**: 過去4週間で反対意見無視が増加傾向にあり、エスカレーション遅延が進行している。
  - **最小シグナル**: 3つのメトリクス（報告遅延、判断分散、反対意見無視）が同時に閾値を超えており、エスカレーション遅延の兆候が明確。

### Run #2（Helm提案反映後 / After）

- 介入内容（アイデア）:
  - 特定のリスクレベル以上のインシデントは、自動的にPMとExecutiveにタグ付けして通知。
  - 「ミスの報告」と「影響範囲の一次評価」をセットで行うテンプレートを導入。
- メトリクス（理想像）:
  - ignored_opposition_count = 0（リスク提起メッセージが速やかに拾われる）
  - 次のRunでは、決定ログに「リスク対応方針」が明記される。
- Findings（期待）:
  - R2_escalation_gap のseverity低下 or 消失。

## Case 3: 外部トレンドチャンスニュース

### 状況（ナラティブ）

- 業界メディアで、自社サービス領域に関する大型トレンドニュースが出る。
- 内容としては「この半年以内に、この分野に投資したプレイヤーが一気にシェア獲得できる」レベルのチャンス。
- 社内では、
  - Slackなどでニュース共有はされる。
  - ただし、「誰がこのチャンスを意思決定テーブルに乗せるか」が曖昧で、アクションに落ちない。

### Run #1（ニュース共有直後 / Before）

- メトリクス（例）:
  - decision_concentration_rate ≒ 0.5（通常業務のまま）
  - kpi_downgrade_count = 0
  - ignored_opposition_count = 0（ここでは主に「チャンス情報の未活用」が問題）
- Findings（期待）:
  - P1_planning_authority_mismatch（計画と権限のミスマッチ）※新パターン
    - 「チャンスに対する意思決定権限が、今の組織構造には存在しない」
- Helm視点:
  - 「誰も悪くないが、チャンスを拾うための**責任者ロール**が設計されていない」ことに着目。

### Run #2（Helm提案反映後 / After）

- 介入内容（アイデア）:
  - 「チャンス検知用の場」を設け、Executive/PM/戦略担当が集まる軽量なレビュー会を定期開催。
  - 既存プロジェクトのKPIとチャンス案件を比較し、資源再配分の優先度をつけるプロセスを追加。
- メトリクス（理想像）:
  - 次回Runでは、当該ニュース関連のプロジェクトが正式な案件として立ち上がる。
  - decision_concentration_rate はむしろ一時的に高まるが、「正しいレイヤーの人」が決めている状態。
- Findings（期待）:
  - P1_planning_authority_mismatch 解消。

---

## まとめ: 3ケースの位置づけ

- Case1（経営会議後）:  
  - **「正当化フェーズ」と「撤退ライン不在」** をHelmが可視化 → 経営アジェンダの再設計へ。
- Case2（重大ミスチャット）:  
  - **「現場止まりのリスク情報」と「責任境界の曖昧さ」** を可視化 → エスカレーション構造の再設計へ。
- Case3（外部トレンド）:  
  - **「チャンスに対する意思決定者不在」** を可視化 → 新たな責任ロールとレビュー場の設計へ。

これら3つを通じて、Helmは「タスクの自動化」ではなく、  
**「組織構造・責任・意思決定パターンをPDCAの対象にするOS」である**ことをデモで示す。

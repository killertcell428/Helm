## Helm 1か月MVP デモ台本（Biz 新規事業ユースケース）

### 0. 導入（口頭）

- 「Helm は、タスクや成果物ではなく、**プロジェクトの進め方そのものが壊れていないか**を見るツールです。」
- 「今日は、健全な新規事業チームと、Zombie Project化しつつあるチームを並べて、Helm がどこを見ているかをお見せします。」

---

### 1. 画面1：ホーム / NWC全体ダッシュボード（`/`）

ナレーション例：

- 「ここがHelmのホーム画面です。舞台は、通信・DX・新規事業を抱える架空企業 NeoWave Communications（NWC）です。」
- 「上のヒーローと3つのカードで、Helmが解決したい **3つの構造的盲点（責任境界の不可視化 / 構造固定化バイアス / 人主語の認知限界）** を一気に示しています。」
- 「少し下にスクロールすると、NWC内のプロジェクト一覧（Healthy / Zombie）と、3つの業務ケースデモ（経営会議後 / 重大ミス / 外部トレンド）が並んでいます。」

操作：

- ブラウザで `/` を開き、ヒーロー＋3つの構造盲点カードを軽く見せる。
- その後、画面中央付近の「NWC内プロジェクトの構造ヘルスチェック」セクションで、`Healthy Project (Biz)` と `Zombie Project (Biz)` の2つが見えている状態を見せる。
- `Zombie Project (Biz)` の「構造アラートを見る」ボタンをクリックして `/projects/project_zombie/alerts` へ進む。

---

### 2. 画面2：Zombie の構造診断一覧（`/projects/project_zombie/alerts`）

ナレーション例：

- 「Zombie Project の構造診断画面です。」
- 「上の段は、Helm が1日の終わりに集計した **構造指標** です。」
  - 「判断の集中度が 70% を超えていて、ほとんどPMだけが決めています。」
  - 「KPI目標の下方修正が3回続いていて、『続ける理由』だけが増えている状態です。」
- 「下の段が、Helm が付けた **構造的なラベル** です。」
  - 「ここでは『判断の先送り・判断集中』『正当化フェーズ』という2つが検知されています。」

操作：

- 指標3つ（判断集中度/KPI下方修正/反対意見無視）を軽く指差し説明。
- Findings の `B1: 正当化フェーズ（Zombie Project化の兆候）` をクリックして詳細へ。

---

### 3. 画面3：正当化フェーズの詳細（`/alerts/project_zombie/B1_justification_phase`）

ナレーション例：

- 「ここでは、Helm が『正当化フェーズ』とラベルを付けた理由を、人間が読める文章で説明しています。」
- 「ポイントは、**何が壊れているかを、KPIや決定ログから構造として説明している**ところです。」
- 「単に『売上が悪い』ではなく、KPI目標を下げ続けて、撤退ラインが一度も議論されていない、という構造の問題を指摘しています。」

操作：

- 説明テキストの重要な1〜2行を読み上げる。
- 下の「この時点での構造指標」も合わせて示し、構造と数字が対応していることを伝える。

次に：

- 「では、この構造が1日の中でどう積み上がっているのかを見てみましょう。」と言って、`/projects/project_zombie/day` に戻る。

---

### 4. 画面4：Zombie の1日タイムライン（`/projects/project_zombie/day`）

ナレーション例：

- 「これは、Zombie Project のある1日の流れを、AM→会議→終業時という3つのフェーズで可視化したものです。」
- 「午前中は、メンバーやPMが普通に仕事をしているだけです。ここではまだ何も壊れていません。」
- 「午後の会議では、『Data担当が撤退を提案しているのに、PMが毎回継続を決めている』ことが分かります。」
- 「終業時に、Helm がこのログを集計して、『判断がPMに集中し、KPIも下方修正を繰り返している。これは正当化フェーズに入っている』と構造的に判断します。」

操作：

- AM・午後のセクションをざっとスクロールし、Dataの反対意見とPMの継続判断を指でなぞる。
- EODセクションで、構造指標とFindingsの日本語ラベルを見せる。

クロージングへのつなぎ：

- 「ここまでが、Helm なしで自然に進んでしまった Zombie Project の1日です。」
- 「次は、同じ枠組みでHealthy Projectを見ると、構造がどう違うかが分かります。」

---

### 5. 画面5：Healthy の1日タイムライン（`/projects/project_healthy/day`）

ナレーション例：

- 「こちらはHealthy Projectです。午前・午後の流れは似ていますが、構造が違います。」
- 「Data担当の意見も意思決定に反映されていて、KPIも下方修正ではなく、目標に向けて改善されています。」
- 「終業時のHelmのチェックでも、構造的な問題は検知されていません。」

操作：

- Zombieとの対比が分かるように、特にEOD部分で「構造指標が健全」「Findingsがゼロ」であることを見せる。

---

### 6. クロージング（口頭）

- 「Helm は、タスクを自動化するツールではありません。」
- 「**プロジェクトがどのように壊れていくか、その構造を可視化し、Executive やPMに『ここがおかしい』と伝えるレイヤー**です。」
- 「3か月版では、ここからさらに『人を呼ぶべきタイミング』や『構造変更の提案』まで踏み込んでいきますが、1か月版ではまず、**構造の壊れ方を見せるところ**までをMVPとして実装しています。」

---

### 7. 将来展望（オプション：時間があれば）

- 「Helmの将来展望は、3つの方向性で整理できます。」
- 「**第一に、横展開です。** Helmは、実現性とインパクトが高い領域を優先して展開します。危機管理、予算配分、組織変革など、暗黙知・政治性が絡むすべての不確実性プロジェクトで組織構造の学習・改善を実現します。」
- 「**第二に、AI主導の企業システムです。** Helmは、新しいヒト×AIの共生関係を実現します。**人は舵を取る。AIは船でありパドルでありレーダーである。** 人は方向性と責任に注力し、AIはアクションを実行します。これは、既存のソリューションにはない、初めてのヒト×AIの共生関係です。」
- 「**第三に、AI会社パッケージです。** Helmは、アイデアから組織実現までの基盤を即座に構築します。**数時間でプロジェクトメンバーと体制や役割を構築、数日でPJは動き出します。** なぜなら、そのロードマップや設計と実作業もAIが提案し人と推進できるためです。」
- 「**Helmのビジョンは、"Helm is where humans steer and AI rows."です。** これは、既存の「人がAIを呼び出す」でも「完全自動化」でもない、**新しいヒト×AIの共生関係**です。」

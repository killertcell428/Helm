# Helm の適応領域と制約

**目的**: Helm が「どこまでを扱うか」「何をしないか」「なぜその範囲でうまくいくか」を、初めて読む人にも伝わる形で定義する。設計・開発・記事・FAQ はこのドキュメントに合わせて整合を取る。

**一文で言うと**:  
Helm は、**特定プロジェクトの定例会議と特定ワークスペースのチャットだけ**を対象に、**判断の遅れ・責任の曖昧さ・プロジェクトが止まりそうな兆候**といった「意思決定の詰まり」を検知し、適切な人に判断を依頼する。1on1・DM・通報窓口などは見ない。人を評価・査定する用途には使わない。

---

## 1. 対象（Helm が扱う範囲）

### 1.1 どのデータを見るか

- **会議**: 特定プロジェクトの**定例会議**（例: 週次経営会議、プロジェクトの進捗会議）。全社の会議を横断して「全部見る」ことはしない。
- **チャット**: 特定**ワークスペース（またはスレッド）のチャット**。どの会議・どのチャットを対象にするかは、導入時に決めてホワイトリストで管理する。
- **資料**: 会議に紐づく会議資料（取り込み可能な範囲で）。分析の補足に使う。

### 1.2 何を検知するか

Helm が検知するのは、**意思決定のプロセスで起きている「詰まり」**です。具体的には次のような状態です。

- **判断の遅れ**: 数字や状況は悪化しているのに、戦略の見直しや撤退の議論が会議で起きていない。
- **責任の曖昧さ**: 「誰がいつ決めるか」がはっきりしておらず、報告やエスカレーションの基準が曖昧。
- **プロジェクト阻害の兆候**: 会議では表に出ていないが、チャットでは「言い出せない」「空気で決まってしまう」といった本音が出ている。

これらを「**意思決定の詰まり**」と呼び、Helm はこの詰まりを検知したときに、あらかじめ決めた**役割（誰が何を決めるか）**と**承認のルール**に従って、適切な人に判断を依頼する。

### 1.3 なぜ「3つの壁」を解消するとよくなると言えるか

Helm が想定しているのは、**定例会議やプロジェクトのチャットで「誰が・いつ・どう判断するか」が問われる場面**です。この場面では、

- 責任が曖昧
- 反対意見が出にくい空気
- 人に頼る判断の限界

の**3つの壁**が、意思決定が進まない主な要因になりやすい、としています。  
だから、「この範囲（定例会議・プロジェクトチャットの意思決定の詰まり）に限定すれば、3つの壁を解消することが効く」と説明しています。  
それ以外の要因（例: 個人の能力、別の業務プロセス）は、Helm の対象外か、今後の拡張として扱います。

---

## 2. 対象外（絶対に見ない・やらない）

### 2.1 見ないデータ・場面

以下は **絶対に対象にしない** と宣言しています。こうすることで「監視が付くと本音を言わなくなるのでは？」という懸念に答えます。

- **1on1**
- **私的なDM（ダイレクトメッセージ）**
- **メンタル相談・労組窓口・内部通報窓口** など、守秘が前提のやりとり

これらはホワイトリストに含めず、取得対象から外す運用にします。

### 2.2 Helm がやらないこと

- **人を評価・査定する**: 会議やチャットの「内容で人を採点する」ツールではない。人事評価・査定には使わない。
- **タスクの自動化や成果物の最適化**: タスク管理ツールや既存のAIエージェントがやるような「作業を代行する」ことは対象外。
- **個人の行動分析**: 誰が何回発言したかで個人を分析するような用途は対象外。
- **ルール違反の監視**: コンプラや規程違反を主に監視するツールではない。「判断が止まっている」状態を検知する。

---

## 3. 制約（なぜその範囲にしているか・何で守るか）

### 3.1 取得範囲の限定

- 対象は「特定プロジェクトの定例会議」「特定ワークスペースのチャット」に限定する。
- 全社の会議・チャットを横断して「全部見る」ことはしない。どの会議・どのスペースを対象にするかは、導入時に決めてホワイトリストで管理する（実装済み）。

### 3.2 同意・告知

- 対象とする会議・チャットには「Helm がログを分析しています」と明示し、ポリシーに同意したメンバーだけが参加する運用を想定する。

### 3.3 データの最小化・短期破棄

- **議事録・チャットの原文は原則保存しない**。保存するのは「リスクの有無」「スコア」「根拠の要約（個人が特定できない形）」など、必要最小限のシグナルだけ。誰が何を言ったか全文を残さない設計（原文はデフォルト7日で破棄、実装済み）。
- 個人名は役職に置き換え、金額や個人を特定できる情報はマスキングする（実装済み）。

### 3.4 見られる人・使われ方の制限

- 閲覧できるのは**ガバナンス担当**などに限定する。人事評価・査定には使わないことを規程で禁止する運用を想定する。
- 誤検知はワンクリックで訂正でき、誰が何を見たかは監査ログで追跡する（実装済み）。重要事項は事前に周知し、必要に応じて労使で協議する運用を想定する。

### 3.5 救済

- 「このアラートは不要だった」をワンクリックでフィードバックでき、その結果をルールや閾値の見直しに反映する（実装済み）。

## 4. 仕組み（どういう設計でうまくいくか）

### 4.1 検知の信頼性

- **ルールとAIの組み合わせ**: 数値条件（ルール）と、複数視点のAI評価（経営者・経営企画・現場・ガバナンスの4つの視点）を組み合わせて、見落としを減らしつつ、根拠を出せるようにしている。
- **証拠の表示**: アラートには「どの発言・いつ」を根拠として表示する。確信度が低いときは「質問として投げる」形にし、重要度に応じて**通知 → レビュー → 承認依頼 → 議題化**の順で段階を上げる。
- **最終判断は人**: AI は「こういう問題がありそう」と提案するまで。承認するか・却下するかは必ず人が決める。

### 4.2 会社ごとの違い（役割・承認のルールの差し替え）

- **誰が何を決めるか**（役割）と**承認の流れ**（誰の承認を経るか）は、JSON で設定する（`config/definitions/` の組織グラフ・RACI・承認フロー）。会社ごとに中身を差し替えられる。
- 決裁者ごとの個性や社風の違いは、この「役割と承認のルールの設定」を変えることで吸収する設計。AI はパターン検知と提案までで、最終判断は人が行う。

### 4.3 因果の範囲の説明

- 検知は相関ベース（「こういう状態のとき、こういうリスクが出やすい」）。改善効果は、検知から経営層の承認までの日数や、経営会議への議題化の有無・日数で測定して検証する設計。
- AI は提案に留め、最終判断は人が行うため、因果の取り違えリスクを抑えている。

## 5. つっかえポイントへの答え方（要約）

| 想定質問 | 答えの要点 |
|----------|------------|
| 監視が付くと正直に話さなくなるのでは？ | 対象を「定例会議・特定チャット」に限定し、1on1・DM・通報窓口は見ないと宣言している。原文は残さず短期で破棄し、同意・告知・誤検知の訂正で「正直に話さなくなる」リスクを抑える設計。 |
| 評価軸の信頼性・なぜAI？・決裁者の個性は？ | ルールと4視点のAIを組み合わせ、根拠を表示。最終判断は人。役割と承認のルールはJSONで差し替え可能なので、会社ごとの違いは設定で吸収。 |
| 3つの壁以外の要因は？ 3つ解消すればなぜよくなる？ | 対象を「定例会議・プロジェクトチャットの意思決定の詰まり」に限定している。この範囲では3つの壁が主因になりやすいため、3つを解消することが効く。それ以外は対象外または今後の拡張。 |

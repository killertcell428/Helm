# 事前ハッカソン向け ギャップ整理 & 想定Q&A

最終更新: 2026-01-21

---

## 1. どこまでできていて、どこからがギャップか

### 1-1. データ面

- 現状
  - デモは NeoWave Communications という仮想企業データ（議事録・チャット・KPI）をベースにしたモック
  - Google Meet / Chat / Drive / Docs との統合は済んでいるが、
    実データ接続は限定的（個人アカウント・PoCレベル想定）
- 本当はこうしたい
  - 特定部署・特定PJを対象に、実会議ログ・実チャットログ・実KPI を利用した PoC を回したい
- 本戦までの案
  - 1部署・1〜3PJをスコープに、Meet/Chat/Drive のいずれか 1〜2 本を実データ接続
  - Firestore 等に「実行履歴・構造スコア・エスカレーション履歴」を蓄積し、
    “Before/After” を定量的に示せるようにする

### 1-2. アルゴリズム・LLM活用

- 現状
  - 構造的問題検知はルールベース + LLM の組み合わせで実装済み
  - 重要度・緊急度・影響範囲、定量メトリクス、マルチ視点評価の枠組みはある
  - ただし、実データに対する「チューニング」はこれから
- 本当はこうしたい
  - 実案件の履歴をもとに、パターンやスコアのしきい値を再定義
  - ロールごとの視点（CxO / 現場PM / TechLead など）をより精緻にモデル化
- 本戦までの案
  - 実データ 3〜5 ケースで「どこで過検知・過少検知しているか」を検証し、
    ルールとプロンプト・スコアリングを再調整

### 1-3. システム面（信頼性・運用）

- 現状
  - FastAPI + ADK + WebSocket で一通りのフローは動作
  - モックフォールバックやエラーハンドリングも最低限入っている
- 本当はこうしたい
  - 本番運用を前提とした監査ログ・アラート・SLO 設計
  - レート制限・キュー・キャッシュによるスケールアウト
- 本戦までの案
  - PoC レベルでは「どのイベントをログに残すか」「誰がどのデータにアクセスしたか」を
    追跡できる監査ログを優先

### 1-4. UX / 導入プロセス

- 現状
  - 「Case1 の理想的なストーリー」はデモとして表現できている
  - 実際の現場での「誰がいつ Helm を起動するか」はまだ仮説レベル
- 本当はこうしたい
  - 具体的な現場ペルソナ（PM / PdM / 経営企画 etc.）ごとの利用シナリオを整理
  - 週次 / 月次 / 四半期ごとの定例にどう組み込むかを設計
- 本戦までの案
  - 1〜2パターンの具体的な導入ストーリー（例: 「四半期経営会議の 1週間前に Helm を起動」）を文章化・図解

## 2. 想定Q&A（突っ込まれたときの回答メモ）

### Q1. 実データで本当に回るイメージはどこまで持てていますか？

- 現時点
  - Meet / Chat / Drive / Docs の実 API 統合は完了しており、
    個人アカウントレベルでの実データ取得は確認済み。
  - ただし、デモでは意図的にモックデータを使い、ストーリーと再現性を優先している。
- 本戦までにやること
  - 1部署・1〜3PJを対象に、実会議ログ・実チャットログを用いた PoC を構築。
  - デモで使っている Case1 フロー（ingest → analyze → escalate → approve → execute）を、
    そのまま実データに対して適用して検証する。
- それ以降の構想
  - 組織横断での運用を想定し、部門ごとに Helm 導入のオンボーディングパスを設計する。

### Q2. 現場導入のボトルネックは何だと思いますか？

- 現時点
  - 最大のボトルネックは「権限とデータアクセス」。
    - 会議ログ・チャットログ・KPI など、センシティブなデータにどこまでアクセスさせるか。
  - 次点で「既存の意思決定プロセスとの整合」。
- 本戦までにやること
  - 小さなスコープ（1部署・1PJ）で、アクセス権限と承認プロセスを明示した PoC を設計。
  - Helm が直接手を出さない領域（最終決裁、対外コミュニケーションなど）を線引きする。
- それ以降の構想
  - RBAC と組織図を取り込んだ「権限付きエスカレーションエンジン」として拡張し、
    コンプラ部門とも連携できる形に育てる。

### Q3. スケールさせるときにボトルネックになるコンポーネントは？

- 現時点
  - ADK エージェントの数と、外部 API 呼び出し（LLM / Google APIs）がボトルネック。
  - 1回の経営会議あたりの負荷は問題ないが、全社展開すると同時実行数が増える。
- 本戦までにやること
  - キューイング（バックグラウンドタスク）とレート制限の設計を明文化。
  - 「1部署・数案件」レベルでの負荷テストを簡易的に実施。
- それ以降の構想
  - 分散キュー（Pub/Sub 等）やキャッシュ層を導入し、大規模組織でも回るアーキに拡張する。

### Q4. セキュリティやコンプラ面はどう考えていますか？

- 現時点
  - 認証は OAuth / サービスアカウントを利用し、Google 側の権限モデルの上で動かす前提。
  - デモ環境では個人アカウント + モックデータのみを使用し、本番データは扱っていない。
- 本戦までにやること
  - PoC スコープで「どのデータを扱うか」「どこまで匿名化・集約するか」を定義。
  - 実行ログ・アクセスログを Firestore 等に残す仕組みを実装し、最低限の監査性を確保。
- それ以降の構想
  - 社内の ISMS / プライバシー規定に合わせたデータマスキング・保持期間ポリシーの設計。

### Q5. 「AIじゃなくてもできるのでは？」という指摘への回答

- 現時点
  - 個々の処理（レポート作成・通知・会議設定）は、確かに既存ワークフローでも可能。
  - しかし Helm が狙っているのは「構造的問題の検知」と「介入タイミングの自動化」。
- 本戦までにやること
  - Before/After の指標（リードタイム・正当化フェーズ検知率・代替案検討率など）を
    数値で示せるように整理。
- それ以降の構想
  - 「どの会議が本当に危険か」「どのプロジェクトを止めるべきか」を
    横断的に示すダッシュボードとして拡張し、「やめる意思決定」の質そのものを上げる。

## 3. まとめ（事前ハッカソンで強調したいこと）

- 今日のデモはモック中心だが、
  - API / サービス層のインターフェース
  - エージェント構成
  - 進捗表示や成果物の流れ
  は本番を前提に設計している。
- 足りていないのは「実データでのチューニング」と「権限・運用の落とし込み」であり、
  - それを 2月の本戦までにどこまで進めるかが勝負どころ。  
  - 事前ハッカソンでは「そこへの道筋と設計の解像度」を見てほしい、というメッセージで臨む。

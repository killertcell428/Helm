最終更新: 2026-01-21

## 1. 課題：研究開発PJの「構造的問題」が見えない

- 研究開発や新規事業のプロジェクトでは、四半期レビューや経営会議がたくさん開かれているが、
  - 会議では「正当化」が続き、
  - チャットでは「本音の不安」が漏れているのに、
  - 構造的な問題（誰に判断が集中しているか・どこでエスカレーションが詰まっているか）が見えない。
- 結果として、
  - 戦略見直しのタイミングを逃し、
  - やめるべきプロジェクトが延命され、
  - 現場のリソースがすり減っていく。

> 「数字は厳しいが、戦略は間違っていないと思う」  
> 「撤退案を口に出す空気ではなかったですね」

こうしたパターンを、人間だけでモニタリングし続けるのはほぼ不可能です。

---

## 2. 解決コンセプト：Helm – 自律改善 AI の「航海士」

- Helm は、会議議事録やチャットログをもとに、
  - 構造的問題（正当化フェーズ・報告遅延・判断集中 など）を検知し、
  - 重要度・緊急度・影響範囲をスコアリングし、
  - 「誰に・どのタイミングで」エスカレーションすべきかを提案する
  AIベースの自律改善プラットフォームです。
- さらに、
  - 次回会議のアジェンダ案
  - 3案比較（継続・縮小・撤退）のドラフト
  - 関係者向けの通知ドラフト
  までを AI が自律的に準備し、「人が本当に悩むべき意思決定」に集中できるようにします。

キーメッセージは  
**「AIが止まるのは、判断が必要な時だけ」** です。

## 3. アーキテクチャ概要

- フロントエンド: Next.js / React / Tailwind  
- バックエンド: FastAPI + ADK（Agent Development Kit）  
- 連携サービス:
  - Google Meet / Chat / Drive / Docs
  - Gemini（Gen AI SDK）による LLM 連携

フローの要約（Case1）:

1. 会議・チャットログを ingest（モック or 実データ）
2. 構造的問題検知エンジンで「正当化フェーズ」などのパターン検出
3. エスカレーション先と理由の自動提案
4. Executive の承認（人間の判断）
5. AI 自律実行（リサーチ・シミュレーション・資料ドラフト・通知ドラフト）
6. 実行結果の表示と、次の PDCA サイクルへ

※ 実装詳細は `Dev/docs/status/CASE1_EXECUTION_FLOW.md` / `PRE_HACKATHON_SCOPE.md` 参照。

## 4. デモ：Case1 – 「戦略見直しの構造的先送り」をどう変えるか

### 4-1. Before（人だけの場合）

- 四半期経営会議の議事録と、その後の経営企画チャットを見ても、
  - KPI 下方修正が続いているのに、
  - 「やめる」「縮小する」という選択肢が議論されていない。
- チャットでは
  - 「やめた方がいいプロジェクトもそろそろ整理しないと」
  - 「撤退案を口に出す空気ではなかった」
  といった不安が共有されているが、経営会議には反映されない。

→ 意思決定リードタイムは 70–85 日、代替案検討率は 10% 未満。

### 4-2. Helm ありの場合のフロー

1. **データ受領**  
   - 会議議事録と会議後チャットを Helm に取り込む（デモではパターン A/B/C を選択可能）。
2. **構造診断**  
   - 構造的問題パターン（例: 正当化フェーズ強め・報告遅延）を検知し、
   - 総合スコア・重要度・緊急度・影響範囲を可視化。
3. **エスカレーション提案**  
   - 「誰に」「どのレベルで」上げるべきかを提案し、
   - 次回会議の強制議題（継続・縮小・撤退の 3案比較）を推奨。
4. **Executive の判断**  
   - Executive が「了承する／一部修正して実行」を選択。
5. **AI 自律実行**  
   - 市場・競合・社内データ（モック）を使った 3案比較シミュレーション、
   - Docs での資料ドラフト生成、
   - 関係者への通知ドラフト作成、
   - 次回会議のアジェンダ案作成までを自律的に実行。
6. **結果とインパクト**  
   - 3 週間後の経営会議では「撤退案」が自然に議題に含まれ、
   - 意思決定リードタイムは 2 日以内に、代替案検討率は 50%以上に改善（フェルミ推定ベース）。

## 5. ロードマップとスコープの線引き

- 今日見せる範囲（デモ）
  - Case1 の一連のフロー（ingest → analyze → escalate → approve → execute → results）
  - モックデータだが、API とサービス層は本番前提のインターフェースで実装済み
  - AI 実行中に「裏で何をしているか」が、エンジニア目線でも違和感ない粒度で可視化

- 2月の本戦までにやる予定
  - 特定部署・特定PJに対する実データ PoC（Meet / Chat / Drive / Docs の一部実接続）
  - Firestore 等への実行履歴・スコアの永続化
  - 実データを使った 3〜5 ケースの検証と、ルール/スコアのチューニング

- それ以降（中期）
  - 組織図・権限モデルを取り込んだエスカレーションエンジン
  - 本番運用を見据えた監査ログ・アラート・スケール設計

この線引きは `PRE_HACKATHON_SCOPE.md` / `DEMO_TO_PROD_SWITCH.md` にも整理済み。

## 6. クロージングメッセージ（事前ハッカソン用）

- Helm は、「プロジェクトがなぜ決まらないのか」という構造的な問いに対して、
  - 会議とチャットから構造的問題を検知し、
  - 正しいタイミングで Executive を呼び出し、
  - そこまでの準備（リサーチ・シミュレーション・資料ドラフト）を自律的に肩代わりする
  「自律改善の航海士」です。
- 事前ハッカソンでは、
  - そのコンセプトとアーキテクチャが十分に筋が良いこと、
  - 本戦までにどこまで実データ検証を進めるかの道筋が描けていること、
  を見ていただきたいと考えています。

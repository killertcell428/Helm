# 1か月本番プレゼン向け：棚卸と計画の前提

**作成日**: 2026年2月21日  
**目的**: 本番プレゼン（約1か月後）に向けて、現状の整理と今後の設計・開発計画を立てるための前提をまとめる。

---

## 1. 前提・制約

| 項目 | 内容 |
|------|------|
| **タイムライン** | 約1か月後に本番プレゼン。その時点で「本格的な実装の状態」で挑みたい。 |
| **本番デプロイ** | 提出済み・審査中のため **Cloud Run / Vercel の本番は変更しない**。新機能・修正は別環境（別サービス／プレゼン用ブランチ／ステージング）で進める。 |
| **記事・動画** | Zenn記事・YouTube動画は完了・投稿済み。コンテンツの大枠は固定し、実装の進捗に応じて記事の「現状の実装状況」だけ必要に応じて更新する想定。 |

---

## 2. 3月向け対応の優先度（順）

### 2.0 優先度の並び

1. **★最優先**: Helmの機能を聞いたときにつっかえるポイントに答えられるようにする → **コア機能の追加実装と拡充**
2. **その範囲で常に使える機能に本番化**: デモ①以外でも、定義した適応領域内でいろんな会議・タスク・チャットの組み合わせでも一定して動くようにする
3. **ADK・ガバナンスの残り、その他**

以下、★の「つっかえポイント」と対応方針を整理する。

---

### 2.1 ★つっかえポイントと対応の方向性

「Helmの機能を聞いたときにつっかえるポイント」に**答えられるようにする**ことを最優先とする。そのために**適応領域**をはっきりさせ、「ここまでを制約とし、こういう仕組みにしているからうまくいきます」と言い切れる状態にする。**詳細は [適応領域と制約](../design/適応領域と制約.md) を参照。**

| つっかえポイント（想定質問） | 対応の方向性 | 必要なもの（設計・実装） |
|----------------------------|--------------|---------------------------|
| **監視が付くと人は正直に話さなくなるのでは？陰で話すようになるリスクをどう対策するの？** | 適応領域・制約を明示する。「監視」ではなく何を目的にどこまで見るか、対象外（1on1・DM・通報窓口等）を宣言し、データ最小化・同意・救済の仕組みで「正直に話さなくなる」リスクを抑える設計であることを説明できるようにする。 | 適応領域ドキュメント（対象／対象外／目的の一文）。[適応領域と制約](../design/適応領域と制約.md) に整理済み。必要ならUI/FAQでの明示。 |
| **経営層・スタッフなどの評価軸は面白いが、信頼性や「なんでそうなの？」がよくわからない。AIでやる理由は？会社ごとに決裁者の個性や特徴が変わるが考慮できているか？** | 評価軸の根拠（なぜ4ロール・ルール×LLMか）、信頼性の出し方（ハイブリッド・証拠表示・誤検知FB）を一言で言えるようにする。決裁者ごとの個性は「役割と承認のルール（RACI・承認フロー）の設定で差し替え可能」「AIはパターン検知と提案まで、最終判断は人」と明示し、会社ごとの差は設定で吸収する設計であることを説明できるようにする。 | 評価の信頼性・「なぜAIか」の説明文（記事・FAQ・スライド用）。役割・承認ルールの差し替えで会社ごとの違いを扱う旨の明文化。必要ならプロンプトや重みの「組織ごとカスタム」の設計。 |
| **意思決定が歩留まりする要因は3つの壁以外にもありそう。この3つの課題を解決したらなぜよくなるの？** | 適応領域を「3つの壁が効きやすい文脈」に限定する。例：定例会議・プロジェクトチャットにおける「誰が・いつ・どう判断するか」の詰まりに限定し、その範囲では3つの壁を解消することが効く理由（根拠・因果の範囲）を短く説明できるようにする。それ以外の要因は「対象外または今後の拡張」と明示する。 | 適応領域の一文定義（どこまでをHelmが扱うか、どこからは扱わないか）。[適応領域と制約](../design/適応領域と制約.md) に記載。3つの壁→改善の因果を「この範囲なら」と限定した説明。 |

**まとめ（目指す状態）**

- **サービスの適応領域**をはっきりさせ（[適応領域と制約](../design/適応領域と制約.md) 参照）、「ここまでをこういう制約にし、こういう仕組みにしているためうまくいきます」と言い切れるようにする。
- 上記ができたうえで、**その範囲で常に使える機能にする**（下記 2.2）。

---

### 2.2 その範囲で常に使える機能に本番化

- いまは**デモ①しか機能していない**状態。
- 上記の適応領域・制約の定義ができた**範囲内**で、いろんな会議内容・タスク・チャットログの組み合わせでも**一定して動く**ようにする。
- デモ専用の固定データではなく、**その定義の範囲で常に使える本番的な機能**に拡充する（別環境で実装・検証。本番デプロイは審査中なので触らない）。

---

### 2.3 その後：ADK・ガバナンスの残り、その他

- ADK Phase2（実API）、ガバナンスまわりの残り（永続化、通知ポリシー等）、docs/future の拡張、その他は上記のあとに回す。

## 3. 今の状況サマリー

### 3.1 完了しているもの

- **コア機能**: データ取り込み（Meet/Chat/資料）→ 検知・評価（ルール＋マルチ視点LLM＋アンサンブル）→ アラート・承認（EscalationEngine、組織グラフ・RACI・承認フローを JSON で設定）→ AI自律実行（ADK Phase1 モック＋フォールバック）、WebSocket 進捗配信
- **ガバナンス・運用**: 認証（API Key＋ロール）、監査ログ・改ざん検証、誤検知フィードバック・精度指標、レート制限・コスト上限、取得範囲・サプレッション、データ保存期間・定期削除、execute 冪等性
- **提出物**: GitHub 公開、Cloud Run / Vercel デプロイ、Zenn記事・YouTube動画 投稿済み

## 4. 実装状況の3区分（プレゼン用の整理）

プレゼンで「どこまで本格実装か」を説明しやすくするための区分。

| 区分 | 内容 | 例 |
|------|------|-----|
| **実装済み** | 本番デプロイに含まれ、そのまま説明できる機能 | ingest / analyze / escalate / approve / execute（モック）、監査・誤検知・保存期間・冪等性、定義ドキュメント駆動 |
| **設計済み・未実装** | 設計書または記事で言及があり、実装すれば「本格化」に直結するもの | ADK Phase2（実API）、永続化（Firestore）、ベンチマーク検証、CalendarAgent、docs/future の4本 |
| **検討中・未設計** | 方針やスコープがまだ固まっていないもの | 3ヶ月以降の運用、他ツール連携、管理UI、OIDC/IAP 本番認証 |

## 5. 記事で触れている「今後やること」のカテゴリ

記事・DEVELOPMENT_STATUS_SUMMARY・NEXT_STEPS からまとめた、**今後やる候補**のカテゴリ。

1. **ADK Phase2（実API統合）**  
   Vertex AI Search、Google Drive、Chat/Gmail など。現在はモック／ドラフト。プレゼンで「ここまで本実装した」と言いやすい。

2. **Case 2 / Case 3**  
   デモシナリオの拡張。プレゼン時のデモの幅を広げる。

3. **永続化**  
   検知記録・エスカレーション履歴の DB/Firestore 保存。記事で「設計段階」と明記されている部分の実装。

4. **会議調整・CalendarAgent**  
   記事では「設計段階」。設計を詰めてから実装するか、プレゼンでは「設計まで」に留めるかの選択あり。

5. **ベンチマーク・評価**  
   BENCHMARK_EVALUATION_DESIGN に沿った、混同行列・アブレーション・最小 N=20 の検証。有効性の「数字」で示すのに有効。

6. **E2E・パフォーマンス**  
   テスト拡充、レスポンス・負荷の見直し。品質・信頼性の説明に使える。

7. **docs/future の4本**  
   オーナーシップ、マルチテナント、ジョブキュー、通知ポリシー。プレゼンでは「設計まで整備済み。実装はロードマップ」と説明可能。

8. **FAQで「設計で詰める」とされていた項目**  
   ADK選定理由の整理、因果と相関の測定設計、モデル劣化防止の頻度・担当、データ境界（マスキング・保持期間の明文化）、社外展開の普遍性など。設計を詰めると記事・FAQの一貫性が増す。

## 8. 計画を立てる際の論点（優先度・スコープ・リスク）

- **本番は触らない**  
  新機能・変更は別環境で行う。プレゼン用デモは「別デプロイ」か「ローカル／ステージング」で見せる前提にする。

- **「本格的な実装」の定義**  
  審査員にどう見せたいかで優先度が変わる。例: 「ADK を実APIで動かした」／「永続化まで入れた」／「ベンチマークで数字を出した」のどれを最優先にするか。

- **設計と実装のバランス**  
  1か月で全部は実装しきれないため、「ここは設計を固めて記事と一致させる」「ここはコードまで書く」を分けるとよい。

- **リスク**  
  実API統合（Vertex AI Search、Chat API 等）は認証・クォータ・エラー処理の工数がかかる。スコープを「1エージェントだけ実API」などに絞る選択肢あり。

## 9. 次のアクション（計画立案へ）

**3月向けの優先度（セクション2）に沿った進め方:**

1. **★適応領域と「つっかえポイント」への回答準備**  
   - 適応領域・制約・仕組みは [適応領域と制約](../design/適応領域と制約.md) にまとめた。  
   - 監視感・評価の信頼性・3つの壁と改善の因果を、上記の表の方向性で説明できるようにする。  
   - 必要に応じて FAQ・スライド用の文言を用意する。

2. **その範囲で常に動く本番化**  
   適応領域が定義できたあと、デモ①以外の会議・タスク・チャットの組み合わせでも、その範囲内で一定して動くように実装・検証する（別環境で。本番は触らない）。

3. **週単位のマイルストーン例**  
   週1＝適応領域・制約の文書化とつっかえポイント回答のドラフト（[適応領域と制約](../design/適応領域と制約.md) 参照）、週2＝評価の信頼性・役割・承認ルールの差し替えの明文化、週3＝本番化のスコープと実装開始、週4＝ADK・ガバナンスの残り or プレゼン資料・デモ手順の確定。

4. **環境方針**  
   プレゼン用デモは別URL（ステージング）で新機能を見せる想定。本番デプロイは変更しない。

5. **記事・FAQ の更新方針**  
   適応領域・制約・「なぜうまくいくか」が固まったら、記事の該当箇所と FAQ を最小限更新する。

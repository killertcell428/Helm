# 事前ハッカソン向けスコープ整理（Case1中心）

最終更新: 2026-01-21

## 1. 実装済み（リアルに動いている領域）

- バックエンド
  - FastAPI ベースの REST API 一式（`backend/main.py`）
  - Google Meet / Chat / Drive / Docs API 統合
    - 実 API ＋ モックのハイブリッド構成（環境変数と認証ファイルの有無で切り替え）
    - エラー時はモックに自動フォールバック
  - 構造的問題検知エンジン
    - ルールベース + LLM（Gemini / Gen AI SDK）連携
    - パターンID、スコア、重要度・緊急度、証拠、定量メトリクスの生成
  - エスカレーション判断エンジン
  - WebSocket 統合（実行進捗）
  - バックグラウンドタスク実行

- フロントエンド
  - Next.js 16 / React / Tailwind ベースの UI
  - Case1, Case2, Case3 デモページ
  - Helm 分析結果の表示（スコア・パターン・証拠・理由）
  - Executive 承認フロー / 次アクション確定フロー
  - AI 実行開始フロー（`execute` + WebSocket 進捗表示）
  - 実行結果画面から Docs / Drive へのリンク表示

- 認証・セキュリティ
  - OAuth（個人アカウント） + サービスアカウント認証
  - モックモードへの自動フォールバック

- テスト
  - ユニットテスト / 統合テスト / 基本的な E2E テスト

## 2. モックだが「現実的」（2〜3ヶ月で到達可能な領域）

### Case1 フロー（今回のデモの中核）

- 会議・チャットデータ
  - NeoWave Communications という仮想企業の経営会議議事録・会議後チャット
  - 「正当化フェーズ」「報告遅延」「健全な議論」の 3 パターン A/B/C
  - 実在しそうな KPI と組織ロールで構成（ただし実データではなくモック）

- 構造診断
  - スコアリング（重要度・緊急度・影響範囲）と定量メトリクス
  - マルチ視点評価（ロール別スコア）の構造

- エスカレーションと Executive 承認
  - 「誰に、どのレベルで」上げるべきかを提案するロジック
  - Executive による approve / modify と、その後の AI 実行の起点

- AI 自律実行（現時点ではモックフロー）
  - 市場・社外データのリサーチ（`GoogleWorkspaceService.research_market_data` モック）
  - 継続・縮小・撤退の 3案シミュレーション（`analyze_data` モック）
  - Docs 資料ドラフト生成（`generate_document` モック）
  - 通知ドラフト作成（`send_notification` モック）
  - WebSocket / ポーリングでの進捗更新と UI 上の 5 ステップ表示

→ これらは **API インターフェースが実運用を前提に切られており、モック → 実データ にそのまま差し替え可能** な範囲。

## 3. 2月の本戦までに実装を狙う領域

- データソース
  - 特定の部署・PJ を対象にした「実会議ログ・実チャットログ」の一部接続
  - Google Meet / Chat / Drive / Docs のうち、1〜2 本を PoC レベルで本番データに接続

- 永続化 / 分析
  - Firestore 等への永続化（最小限のスキーマ）
    - 実行履歴・構造スコア・エスカレーション履歴の保存
  - 実データを使った 3〜5 ケース分の分析シナリオ

- テスト・品質
  - WebSocket 統合を含む E2E テストの拡充
  - LLM フォールバックパスを含むテストの整理

## 4. それ以降（構想〜中期ロードマップ）

- 組織・権限モデル
  - 役職・組織図・責任範囲をモデリングした「エスカレーション先の自動決定」
  - 権限に応じた通知・編集範囲の制御

- 信頼性・運用
  - 本番運用を前提とした監査ログ・アラート・SLO 設計
  - レート制限・キューイング・キャッシュ層によるスケーラビリティ確保

- UX / 導入

  - 日常的な使われ方（週次・月次レビュー）に合わせた画面・通知設計

## Helm 3パターン PDCA デモ台本（Bizユースケース）

### 共通メッセージ

- 「Helm はタスクや成果物ではなく、**組織の進め方そのものが壊れていないか**をチェックするOSです。」
- 「今日は、3つの現実的なシーンで『人だけだと上長相談に時間がかかる・手遅れになる』ところを、Helmがどう支えるかを見ていきます。」

---

## Case1: 経営会議後（正当化フェーズ）

### Run #1（会議直後 / state=data_received）

画面: `/cases/1/run/1?state=data_received`

- 「四半期の経営会議が終わった直後の状態です。」
- 「成長率の未達とCAC悪化が報告されましたが、『もう1四半期様子を見る』となっていて、撤退やピボットは議題に上っていません。」
- 「実デモ用テキストとして、CFO・CEO・各本部長の発言と、会議後Slackのログが画面上部に並んでいます。」
- 「ここでは、まだHelmは『データ受領直後』なので、構造指標は計算していません。」
  - 次のステップに進むことで、Helm解析の結果に切り替わります。

### Run #2（Helm解析完了 → ロール判断 → 構造介入後 / state=helm_analyzed → roles_deciding → next_action_decided）

画面: `/cases/1/run/1?state=helm_analyzed` → `/cases/1/run/1?state=roles_deciding` → `/cases/1/run/2?state=next_action_decided`

- 「2ステップ目で、Helmの構造診断が表示されます。」
- 「画面右側には『KPI悪化認識あり / 戦略変更議論なし / 撤退選択肢0件』といった構造指標がカードで並び、下に『正当化フェーズ』の自然文説明が出ています。」
- 「3ステップ目（roles_deciding）では、Executive の判断セクションが強調され、Helmからの提案（撤退案の提案責任者を任命 / KPIトリガー設定など）がボタン付きで表示されます。」
- 「最後に Run2 / next_action_decided に切り替えると、『撤退案スライドが議題に含まれた』『投資凍結オプションにCFOが言及』といった After の世界がカードで表示され、構造スコア改善の数字が出ます。」
- 「Helmの役割は、『数字が悪い』と言うことではなく、『会議の構造がこのままだと手遅れになる』と教えることです。」

## Case2: 重大ミスのエスカレーションチャット

### Run #1（現場チャットのみ / state=data_received）

画面: `/cases/2/run/1?state=data_received`

- 「次は、スタッフの重大ミスが発覚したときのチャットです。」
- 「画面上部には、実際のSlackログ（マーケ担当・運用担当・運用リーダーの会話）がそのままタイムライン形式で表示されています。」
- 「『影響かなり出そうですよね…』『部長に上げるほどかは微妙』といった、人間なら『これはやばいけど、どこまで上げるべきか迷う』典型的な文脈を、あえてそのまま見せています。」
- 「この段階ではまだHelmは解析前で、下の構造指標エリアは空の状態です。」

### Run #2（Helm緊急診断 → ロール判断 → 構造介入後 / state=helm_analyzed → roles_deciding → next_action_decided）

画面: `/cases/2/run/1?state=helm_analyzed` → `/cases/2/run/1?state=roles_deciding` → `/cases/2/run/2?state=next_action_decided`

- 「Helm解析ステップでは、『現場リスク認識：高 / 上位共有：弱い / レビュー権限：不在』という3つのカードが出てきます。」
- 「その下に『リスクは現場で認識されていますが、誰がどこまで判断するかが定義されていない』という Finding の自然文が表示されます。」
- 「ロール判断ステップでは、Executive 用のセクションに『金額・顧客影響数でエスカレーション条件を定義 / CxO直結ルートを明示 / 初動24hで影響試算』といったチェックリストが出てきます。」
- 「最後に Run2 / next_action_decided に切り替えると、『CFO直結ルート実装 / 影響試算の24h自動化 / 再発防止ガイド配信』など、構造介入後の状態がメトリクスカードとして見えます。」
- 「Helmはミスを防ぐのではなく、**ミスが起きたときに組織がどう壊れないようにするか**を設計する役割です。」

## Case3: 外部トレンドチャンスニュース

### Run #1（ニュース共有のみ / state=data_received）

画面: `/cases/3/run/1?state=data_received`

- 「最後は、外部トレンドのチャンスが出たときのケースです。」
- 「画面上部には、競合C社のニュース見出しと、社内横断Slackのログ（『これうちのIoTと相性いい』『PoCなら行けそう』『ただ今期は厳しい』）が並んでいます。」
- 「その下に簡易な企画会議メモとして『面白いテーマ / 既存施策との優先度整理が必要 / 誰が最終判断するかは次回検討』と書かれています。」
- 「この段階では、チャンス認識は高いものの、資源再配分の決定者も決定期限も存在していません。」

### Run #2（Helm機会分析 → ロール判断 → 構造介入後 / state=helm_analyzed → roles_deciding → next_action_decided）

画面: `/cases/3/run/1?state=helm_analyzed` → `/cases/3/run/1?state=roles_deciding` → `/cases/3/run/2?state=next_action_decided`

- 「Helm解析ステップでは、『チャンス認識：高 / 資源再配分決定者：不在 / 決定期限：なし』というカードが出ます。」
- 「その下に『機会は認識されているが、計画と権限の構造が一致していない』という Finding の自然文が表示されます。」
- 「ロール判断ステップでは、Executive 向けに『CEO直下で期間限定の資源再配分権限を付与 / やらない施策のリスト化 / 30日以内のGo/No-Go期限設定』という提案が並びます。」
- 「Run2 / next_action_decided では、『新規事業本部長に決定権委譲 / 既存PJ2本停止 / Go/No-Go会議を30日後に設定』といった After 状態がカードで表示され、意思決定スピードの改善（3ヶ月→2日）が示されます。」
- 「Helmは、チャンスそのものを見つけるのではなく、**チャンスを拾える組織構造かどうか**を監視します。」

## クロージング

- 「3つのケースに共通しているのは、問題が『タスクの出来/不出来』ではなく、**責任・判断・エスカレーションの構造**にあるということです。」
- 「Helmは、AIが仕事を奪うのではなく、**人とAIでできた組織が壊れないようにするOS**です。」
- 「本戦の3か月版では、今日お見せしたミニPDCAを、実際の会議ログ・チャット・ニュースから自動で構造イベントに落とし込み、次のアクション提案までを自律的に回していきます。」

## 将来展望（オプション：時間があれば）

- 「Helmの将来展望は、3つの方向性で整理できます。」
- 「**第一に、横展開です。** Helmは、実現性とインパクトが高い領域を優先して展開します。危機管理、予算配分、組織変革など、暗黙知・政治性が絡むすべての不確実性プロジェクトで組織構造の学習・改善を実現します。」
- 「**第二に、AI主導の企業システムです。** Helmは、新しいヒト×AIの共生関係を実現します。**人は舵を取る。AIは船でありパドルでありレーダーである。** 人は方向性と責任に注力し、AIはアクションを実行します。これは、既存のソリューションにはない、初めてのヒト×AIの共生関係です。」
- 「**第三に、AI会社パッケージです。** Helmは、アイデアから組織実現までの基盤を即座に構築します。**数時間でプロジェクトメンバーと体制や役割を構築、数日でPJは動き出します。** なぜなら、そのロードマップや設計と実作業もAIが提案し人と推進できるためです。」
- 「**Helmのビジョンは、"Helm is where humans steer and AI rows."です。** これは、既存の「人がAIを呼び出す」でも「完全自動化」でもない、**新しいヒト×AIの共生関係**です。」

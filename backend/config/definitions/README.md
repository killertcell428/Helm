# 定義ドキュメント（組織グラフ・RACI・承認フロー）

実務で運用可能なガバナンス定義です。業種・規模に応じてカスタマイズしてください。

## アラートパターン設計方針

### 1. パターン分類（実務ベース）

| カテゴリ | pattern_id | 説明 | 推奨承認 |
|----------|------------|------|----------|
| **戦略・投資** | B1_正当化フェーズ | KPI悪化が続くのに戦略変更・撤退議論がない | 多段階（CFO→経営層） |
| **意思決定** | B2_判断集中 | 意思決定が特定人物に集中している | 単一承認 |
| **意思決定** | B3_反対意見無視 | 反対意見が適切に反映されていない | 単一承認 |
| **ガバナンス** | ES1_報告遅延 | リスク認識があるが経営層への報告が遅延 | 報告フロー（経営企画→経営層） |
| **ガバナンス** | ES2_エスカレーション不明確 | エスカレーション基準・責任の所在が曖昧 | 単一承認 |
| **フォールバック** | default | 上記に該当しないケース | 単一承認 |

### 2. RACI設計の原則

- **R（Responsible）**: 実行責任者。エスカレーション先のプライマリ。
- **A（Accountable）**: 説明責任者。最終承認・判断を担う。
- **C（Consulted）**: 相談先。意思決定前に意見を求める。
- **I（Informed）**: 報告先。事後的に共有する。

### 3. 承認フローの使い分け

- **single_approval**: リスクが低〜中程度。1名承認で完結。
- **investment_strategic**: 投資・戦略判断。CFOレビュー後に経営層承認。
- **compliance_reporting**: 報告遅延・コンプライアンス懸念。経営企画確認後に経営層承認。

### 4. カスタマイズのヒント

- **事業部門が多い場合**: `org_graph` に `role_bu_head` 等を追加し、部門別エスカレーションを定義
- **監査・法務が独立組織の場合**: RACI の C に `role_legal`, `role_audit` を追加
- **上場企業**: 内部統制・四半期報告を考慮したフローを追加

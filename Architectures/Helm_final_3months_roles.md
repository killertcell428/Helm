## Helm 業務体験デザイン（3か月後・本番大会版）

### 0. 前提

- 本番時点では：
  - 1か月版の体験（構造の異常検知＋Executive 呼び出し）が **安定稼働**
  - さらに、**一部の構造変更は実際にシステム上で反映される**
  - Biz ユースケース（新規事業）は「本命デモ」としてフル体験
  - R&D ユースケースは「同じ枠組みで動く」ことを一部デモ

---

## 1. 役割モデル（本番版）

### 1.1 人側の役割

- **Executive（経営層 / 事業責任者 / 研究PI）**
  - Helmから「呼ばれる」立場であり、  
    構造変更の **承認者 / 拒否者** として明確に位置付けられる。
  - 「どこまでをAIに任せ、どこからを人が決めるか」の境界線を握る。

- **PM / チームリーダー**
  - プロジェクト運営の実務責任者。
  - Helm の提案を踏まえつつ、会議設計やKPI設計を調整する。
  - 「Helmをプロジェクト運営にどう組み込むか」をデザインする役割。

- **チームメンバー**
  - 通常業務を行いつつ、  
    Helm が提案した構造変更（例: レビュー体制の変更）に従って動く。

### 1.2 AI / システム側の役割

- **AI Agents（Biz / R&D）**
  - 会議メモや実験ノートから、  
    **「誰が何を主張し、何が採用/却下されたか」** を整理してログ化。
  - KPIの変動や、撤退/継続判断の履歴も記録。

- **Helm Core**
  - **「構造の健康診断」＋「構造の処方箋」＋「処方の実行」** を担う。
  - 一部の構造変更は自動・半自動で反映し、  
    その履歴を残すことで「組織がどう学習したか」を見せる。

## 2. 1サイクルの業務体験（Bizユースケース：本命デモ）

### 2.1 Run #1：従来のやり方＋Helm観測

1. **通常業務**
   - チームはこれまで通り、新規事業のKPIを追いかける。
   - 会議では、「もう少し続けよう」「機能を足せばいけるかも」といった議論が続く。

2. **ログの蓄積**
   - AI Agents が会議メモ・KPIの変化を解析し、  
     - KPI目標の再設定（下方修正）
     - 主な発言者
     - 反対意見の有無  
     を BigQuery に記録。

3. **Helm による構造分析**
   - Helm は「正当化フェーズ」や「Zombie Project」パターンを検出し、
     - 「このプロジェクトでは、撤退ラインが一度も議論されていません」
     - 「KPIが3回下方修正されました」  
     といったフィードバックを生成。

4. **Executive へのアラート**
   - Executive のダッシュボードに、赤いステータスでアラートが上がる。
   - ここまでは1か月版と近いが、本番版では **次のステップとして構造変更案が具体化される**。

---

### 2.2 Executive の体験：構造変更の承認

1. **アラート詳細画面を開く**
   - 「この3ヶ月で、KPI目標が◯回下方修正されました」
   - 「意思決定者が常に同じ人物（PM）であり、反対意見が複数回却下されています」
   - Helm が「構造的に何が起きているか」を文章＋簡易チャートで説明。

2. **Helm からの「構造介入案」を確認**
   - 例：
     - 次回の四半期レビューを「撤退基準を必ず議題に含むボードレビュー」として設定
     - 事業継続の最終承認者を Executive に変更し、  
       PM の単独決定をやめる

3. **Before/After のプレビューを見る**
   - 組織グラフの Before/After を UI 上で確認：
     - Before: PM → （直で）継続/撤退決定
     - After: PM → Helm がフラグ → Executive/ボードが最終決定

4. **承認 or 却下**
   - Executive は「この構造変更を適用する」ボタンを押すことで、
     - Firestore 上の組織グラフが更新され
     - 「誰がどの判断をどの責任で行うか」がシステム上で変わる

→ Executive にとっての体験は、  
**「これまで暗黙だった“責任の構造”を、Helm と一緒に設計し直す」** ことになる。

---

### 2.3 Run #2：構造変更後の世界

1. **新しい意思決定フローでの会議**
   - 次の四半期レビューは、
     - Executive も参加する
     - 会議アジェンダに「撤退/ピボット条件」が明示される

2. **会議の中での変化**
   - PM は「Helmからの提案で、今回は撤退基準も一緒に見ます」と説明。
   - データ担当が、「成功確率が低い」「他案件に人を移した方がいい」といった意見を出す。
   - Executive が最終判断を下す。

3. **Helm の記録**
   - Helm は
     - 誰が最終判断をしたか
     - 撤退/継続のどちらが選ばれたか
     - それに至る議論のパターン
     を再度ログとして記録。

4. **Before/After レポート**
   - ダッシュボード上で：
     - 意思決定リードタイム：7日 → 48時間
     - 代替案の検討率：20% → 65%
     - Executive 参加率：0% → 100%（特定フェーズ）  
     などの指標が Before/After で比較できる。

→ ビジネスユーザーは、  
**「Helmを入れると、会議の空気と責任の置き方が変わり、その結果として数字の動きも変わる」** というストーリーを体験する。

---

### 2.4 本番当日の「LIVE デモ体験」

- 会場では、あらかじめ Helm が起動した状態で、  
  審査員や登壇者がコメントを投げかける。

- 入力パターンは2種類：
  1. **音声入力**  
     - マイクで「このプロジェクト、本当に続けるべきですか？」などと質問  
     - 音声は Speech-to-Text でテキスト化され、Helm に渡る
  2. **テキスト入力（フォールバック）**  
     - 同じ内容を、LIVE セッション画面のテキストボックスにそのまま入力

- Helm が行うこと：
  - その発言を「新しいフィードバック／反対意見」として解釈
  - 直近のログ＋この新発言を踏まえて、構造を再評価
  - 「今この場で取るべきアクション」を 1〜2個、即時に提示

- 例：  
  > 「今のご質問を踏まえると、  
  >  この事業は当初の成功条件を満たしていません。  
  >  次のアクションとして、  
  >  1) 撤退ラインの再確認、2) 人材を別プロジェクトへ再配置する案の検討  
  >  をこの場でボードに諮ることを提案します。」

→ もし音声がうまく動かなくても、  
**「発言内容をテキストで打ち込んで同じ反応を返す」** ことで、  
LIVE 体験の本質（その場で構造を評価し、次アクションを提案する）は守られる。

## 3. R&D ユースケースでの体験（ライト版）

1. **研究チームの現状**
   - 仮説Xの実験が2年目に突入
   - 失敗が増えているのに、「次の実験を回そう」が続いている

2. **Helmの指摘**
   - 「仮説検証フェーズから、仮説維持フェーズに移行している兆候があります」
   - 「データ担当の否定的コメントが3回連続で反映されていません」

3. **構造介入案（例）**
   - 「次回のラボミーティングで、仮説Xをゼロベースで見直すレビューを設定してください」
   - 「仮説の継続/中止判断に、外部アドバイザーを1名追加することを推奨します」

4. **PIの体験**
   - 「自分たちが惰性で続けていたかもしれない」という気付きを得る
   - ただし、Bizよりは「提案＋可視化」が中心で、  
     構造の自動変更は最小限に留める（研究現場の裁量を尊重）

---

## 4. PM / チームリーダーの体験（本番版）

### 4.1 「Helm付きプロジェクト運営」への移行

- PM は Helm ダッシュボードを定期的に確認し：
  - 会議設計の見直し
  - レビューサイクルの調整
  - KPI 設計の見直し  
  を行う。

- 具体的には：
  - 「このプロジェクトは今、正当化フェーズに入りかけている」  
  → 次の会議のアジェンダに「撤退候補の洗い出し」を追加

### 4.2 プロジェクト横断での学び

- Helm は複数プロジェクトのログを見ているため、
  - 「過去にうまくいった撤退判断パターン」
  - 「構造変更後に改善したケース」
  を PM に共有できる。

→ PM にとって Helm は、  
**「プロジェクト運営の暗黙知を横断的に教えてくれる“構造コーチ”」** になる。

---

## 5. チームメンバーの体験（本番版）

- 日常業務は変わらないが、次のような変化を感じる：
  - 会議アジェンダに「撤退/ピボット」「代替案」が定期的に入るようになる
  - データ担当や現場担当の意見が、以前より議論に上るようになる
  - 結果として、「なんとなく続ける」プロジェクトが減る

→ メンバーにとって、  
**「組織の空気が変わった。その裏にHelmがいる」** という位置づけ。

---

## 6. 本番大会で伝えたい「業務体験としての新規性」

- **AIは仕事を奪わない。  
  AIは「責任の設計」と「会議の空気」を変える。**

- **Helm導入前後で変わるのは、成果物そのものではなく、  
  「誰が・どのタイミングで・どの責任で決めるか」という構造。**

- ビジネスユーザーにとっては：
  - 「AIにプロジェクトを任せる」のではなく、
  - 「プロジェクトの壊れ方をAIと一緒に監視し、手遅れ前に手を打つ」体験。
